### ข้อ 1.2 (10 คะแนน)

CI/CD pipeline ของระบบนี้ควรมี stage อะไรบ้างเรียงตามลำดับ (ตั้งแต่ developer push code จนถึง deploy)? อธิบายว่า **แต่ละ stage ทำอะไร** และ **ทำไมต้องเรียงลำดับนี้**

> **ตอบ:** _[ออกแบบ Pipeline โดยใช้หลักการ "Fail-Fast"   
  Stage 1: Checkout Code Jenkins จะทำการดึงซอร์สโค้ดล่าสุดจาก Git Repository หลังจากได้รับสัญญาณ Webhook เมื่อมีการ Merge โค้ดเข้าสู่สายหลัก  
  ทำไมต้องอยู่ลำดับนี้: เป็นจุดเริ่มต้นที่จำเป็นที่สุด เพราะหากไม่มีตัวโค้ดล่าสุด ก็จะไม่สามารถนำไปประมวลผลหรือทดสอบในขั้นตอนอื่นได้เลย  
  Stage 2: Code Linting & Static Code Analysis ใช้เครื่องมือตรวจสอบความสะอาดของโค้ด Python เช่น Flake8 หรือ Black เพื่อเช็ก Syntax, ตรวจสอบ Format โค้ดให้เป็นมาตรฐานเดียวกัน  
  ทำไมต้องอยู่ลำดับนี้: เป็นขั้นตอนที่ทำงานเร็วที่สุด (ใช้เวลาเป็นวินาที) และไม่ต้องอาศัยสภาพแวดล้อมจำลองใดๆ หาก Dev พิมพ์โค้ดผิดฟอร์แมตหรือเขียน Syntax พัง ระบบจะแจ้งเตือนและหยุดทำงานทันทีโดยไม่ต้องเสียเวลาไปรันเทสใหญ่  
  Stage 3: Unit Testing เพื่อตรวจสอบ Logic ภายในแอปพลิเคชัน Flask ว่าฟังก์ชันต่างๆ ยังทำงานได้ถูกต้องตามที่ออกแบบไว้   
  ทำไมต้องอยู่ลำดับนี้: ทำหลังจากจัดฟอร์แมตโค้ดเสร็จ เพื่อพิสูจน์ว่า Logic ของแอปพลิเคชันไม่มีบั๊กและทำงานได้ถูกต้องตามความต้องการ ก่อนที่จะนำโค้ดนี้ไปทำเป็น Container Image ในขั้นตอนต่อไป  
  Stage 4: Build Docker Image ทำการประกอบร่าง (Package) แอปพลิเคชัน Flask, Dependencies ทั้งหมด (จาก requirements.txt) และสภาพแวดล้อมที่จำเป็นขึ้นมาเป็น Docker Image ตามคำสั่งที่เขียนไว้ใน Dockerfile  
  ทำไมต้องอยู่ลำดับนี้: ควรทำหลังจากที่มั่นใจว่าโค้ดไม่มีบั๊กพื้นฐานแล้ว (ผ่าน Stage 2 และ 3) เนื่องจากกระบวนการ Build Image ค่อนข้างใช้เวลาและทรัพยากรเครื่อง (Compute Resource) หากโค้ดพังตั้งแต่ Logic จะได้ไม่ต้องเสียเวลา Build ให้เปลือง Resource  
  Stage 5: Security Scanning (Vulnerability Scan) ใช้เครื่องมือเช่น Trivy หรือ Clair สแกนเข้าไปใน Docker Image ที่เพิ่ง Build เสร็จ เพื่อตรวจหาช่องโหว่ด้านความปลอดภัย (Vulnerabilities) ในระดับ OS Packages และ Library ต่างๆ ที่ดึงมาใช้  
  ทำไมต้องอยู่ลำดับนี้: ต้องทำทันทีหลังจากที่ได้ Docker Image มา เพื่อคัดกรองความปลอดภัยให้มั่นใจ 100% ว่าจะไม่มีโค้ดหรือแอปพลิเคชันที่มีช่องโหว่อันตรายหลุดรอดขึ้นไปรันบน Production Server  
  Stage 6: Push Image to Registry นำ Docker Image ที่ผ่านการตรวจสอบความปลอดภัยเรียบร้อยแล้ว อัปโหลด (Push) ไปเก็บไว้ที่ Private Docker Registry ส่วนกลางขององค์กร โดยติดแท็กเวอร์ชันอย่างชัดเจน (เช่น เลข Git Commit SHA) เพื่อให้ Kubernetes สามารถเข้ามาดึงไปใช้ได้  
  ทำไมต้องอยู่ลำดับนี้: เพราะนี่คือขั้นตอนสุดท้ายของการเตรียม Artifact (ตัวแอปที่พร้อมรัน) ก่อนที่จะสั่ง Deploy จริง Image ต้องถูกเก็บไว้ในที่ที่ปลอดภัยและเข้าถึงได้จากภายนอกก่อน  
  Stage 7: Deploy to Kubernetes Cluster (Production) Jenkins จะส่งคำสั่งไปอัปเดตไฟล์คอนฟิก (Kubernetes Manifests) เพื่อเปลี่ยนเวอร์ชัน Image เป็นตัวล่าสุด และสั่งให้ K8s ทำการ Rolling Update ค่อยๆ ทยอยเปลี่ยน Pod ของ Flask ทีละตัว เพื่อสลับรับทราฟฟิกของผู้ใช้อย่างต่อเนื่องแบบไร้รอยต่อ (Zero-Downtime)  
  ทำไมต้องอยู่ลำดับนี้: เป็นขั้นตอนสุดท้ายที่มีความสำคัญที่สุดและใช้ความเสี่ยงสูงที่สุด จึงถูกวางไว้หลังสุดหลังจากที่ตัวแอปพลิเคชันผ่านการทดสอบทั้งเรื่อง คุณภาพ ความถูกต้อง และความปลอดภัยมาอย่างครบถ้วนแล้ว เพื่อให้มั่นใจได้ว่าระบบใหม่ที่จะขึ้นไปรันจะไม่ทำให้ระบบล่มอย่างแน่นอน]_

## ส่วนที่ 2: Docker & Containerization (20 คะแนน)

### ข้อ 2.1 (7 คะแนน)

อธิบายว่า **container ต่างจาก virtual machine (VM) อย่างไร** และการ containerize แอป Flask + MySQL ช่วยแก้ปัญหา "บนเครื่องผมรันได้แต่บน server รันไม่ได้" ได้อย่างไร

> **ตอบ:** _[ความแตกต่างระหว่าง Container กับ Virtual Machine (VM):
Virtual Machine (VM): เป็นการจำลองในระดับฮาร์ดแวร์ (Hardware Virtualization) ผ่าน Hypervisor โดยแต่ละ VM จะต้องมี Guest OS (ระบบปฏิบัติการเต็มรูปแบบ) เป็นของตัวเอง ทำให้กินทรัพยากร (CPU, RAM, Disk) สูงมาก บูตระบบช้า (เป็นนาที) และมีขนาดใหญ่ (หลาย GB)
Container: เป็นการจำลองในระดับระบบปฏิบัติการ (OS Virtualization) โดยแชร์ Host OS Kernel ร่วมกัน และใช้ฟีเจอร์ของ Linux (เช่น Namespaces, Cgroups) ในการแยกสภาวะแวดล้อม (Isolation) ทำให้ไม่ต้องมี Guest OS ส่งผลให้กินทรัพยากรต่ำมาก บูตเร็วในระดับวินาที และมีขนาดเล็กมาก (ระดับ MB)   
  การแก้ปัญหา "บนเครื่องผมรันได้แต่บน server รันไม่ได้" ด้วย Portability:
ปัญหานี้มักเกิดจากสภาพแวดล้อม (Environment Mis-match) เช่น เวอร์ชันของ Python หรือ Library ในเครื่อง Dev ไม่ตรงกับบน Server
การทำ Containerize จะทำการมัดรวม (Package) ตัวแอปพลิเคชัน Flask, ซอร์สโค้ด, Runtime (Python 3.11), Dependencies ทั้งหมด (requirements.txt) และ Environment Variables ที่จำเป็น เข้าไว้ด้วยกันใน Docker Image เพียงหนึ่งเดียว ทำให้มั่นใจได้ว่า สภาพแวดล้อมที่แอปพลิเคชันใช้จะถูกล็อกไว้คงที่ ไม่ว่าจะนำ Image นี้ไปรันบนคอมพิวเตอร์ของ Dev คนไหน หรือบน On-premise Server เครื่องใดก็ตาม ระบบจะทำงานได้ผลลัพธ์เหมือนกัน 100%]_

### ข้อ 2.2 (7 คะแนน)

ทำไมในระบบ production **ไม่ควรใช้ Docker image tag `latest`**? ถ้าไม่ใช้ `latest` ควรใช้รูปแบบ tag แบบไหนแทน และมีผลต่อการ rollback อย่างไร?

> **ตอบ:** _[แท็ก :latest เป็นแท็กที่แปรเปลี่ยนได้ (Mutable) เมื่อมีการ Build Image ใหม่และไม่ได้ระบุแท็ก ตัว Docker จะย้ายแท็ก :latest ไปชี้ที่ Image ล่าสุดนั้นอัตโนมัติ ทำให้เกิดความเสี่ยงสูงมากในระบบ Production เพราะเราจะไม่สามารถตรวจสอบย้อนกลับได้เลยว่า คอนเทนเนอร์ที่รันอยู่จริง ๆ มาจากซอร์สโค้ด Commit ไหน และหาก Kubernetes มีการสั่ง Restart Pod หรือสั่งขยายโหลด (Scale Out) มันอาจจะไปดึง Image ตัวล่าสุดที่เดฟเพิ่ง Push ขึ้นไปและยังไม่ได้ทดสอบมาทำงาน ส่งผลให้ระบบพังได้โดยไม่รู้ตัว  
รูปแบบ Tag ที่ควรใช้แทน:
ควรใช้แท็กแบบเจาะจงและไม่เปลี่ยนแปลง (Immutable) เช่น Git Commit SHA Short (เช่น :c3f4e5a) เพื่อผูกตรงกับประวัติใน Git หรือใช้ Semantic Versioning (เช่น :v1.2.0)  
ผลต่อการ Rollback (การย้อนกลับระบบ):
หากเกิดปัญหาระบบพังหลังจาก Deploy เวอร์ชันใหม่ การใช้แท็กเจาะจงจะช่วยให้ทีม Ops สามารถสั่ง Rollback ด้วยการแก้ Config ใน Kubernetes ให้กลับไปชี้ที่แท็กเวอร์ชันเก่า (เช่น :v1.1.9) ได้ทันที K8s จะดึง Image ตัวเดิมที่เคยทำงานได้เสถียรกลับมาทำงานได้อย่างแม่นยำ
ในทางกลับกัน หากใช้ :latest ตัว K8s จะไม่สามารถแยกแยะความต่างของ Image ได้เลย (เพราะชื่อแท็กเดิม) ทำให้ไม่สามารถสั่ง Rollback ระบบได้อย่างถูกต้องและปลอดภัย]_

หลักการ: แยกส่วนที่เปลี่ยนน้อย (การลง Dependencies/Libraries) ไว้ด้านบน และเอาส่วนที่เปลี่ยนบ่อย (โค้ดแอปพลิเคชันที่ Dev ต้องแก้ทุกวัน) ไว้ด้านล่างสุด  https://raw.githubusercontent.com/OnpreeyaMi/final_severless/main/image-1.png]_

## ส่วนที่ 3: Infrastructure as Code (20 คะแนน)

### ข้อ 3.1 (8 คะแนน)

**Terraform** กับ **Ansible** ต่างกันอย่างไร? (ตอบในแง่ declarative vs imperative และ provisioning vs configuration management)

> **ตอบ:** _[Declarative vs Imperative (รูปแบบการเขียนโค้ด):  
Terraform (Declarative - กำหนดเป้าหมาย): ผู้ใช้เพียงแค่เขียนอธิบาย "ผลลัพธ์สุดท้ายที่ต้องการ" (Desired State) ลงในไฟล์คอนฟิก (เช่น ต้องการสร้าง VM 3 เครื่อง) โดยไม่ต้องสนใจวิธีการสร้าง ตัว Terraform จะเปรียบเทียบสถานะปัจจุบันกับเป้าหมาย แล้วหาวิธีการไปให้ถึงจุดนั้นเองอัตโนมัติ
Ansible (Imperative - กำหนดขั้นตอน): ผู้ใช้ต้องเขียนคำสั่งระบุ "ลำดับขั้นตอนในการทำงาน" ทีละสเต็ป (Step-by-Step / Task) เช่น เริ่มจากสเต็ป 1 โหลดแพ็กเกจ, สเต็ป 2 ทำการติดตั้ง, สเต็ป 3 เปิดบริการ (แม้ว่าตัวโมดูลของ Ansible จะถูกออกแบบให้ทำงานแบบกำหนดผลลัพธ์ในตัวได้ แต่รูปแบบโครงสร้างหลัก (Playbook) ยังดำเนินตามแนวทางเป็นลำดับขั้นตอนเป็นหลัก)  
Provisioning vs Configuration Management (หน้าที่หลัก):  
Terraform (Infrastructure Provisioning): มุ่งเน้นการจัดเตรียมและสร้างระบบโครงสร้างพื้นฐานขึ้นมาใหม่จากความว่างเปล่า เช่น การสั่งเปิดเครื่อง Server, สร้าง Network, สร้าง Storage หรือกำหนดสิทธิ์บนระบบ Cloud/On-Premise ผ่าน API
Ansible (Configuration Management): มุ่งเน้นการปรับแต่งและจัดการระบบภายใน Server หลังจากที่ตัวเครื่องถูกสร้างขึ้นมาแล้ว เช่น การติดตั้งแพ็กเกจซอฟต์แวร์, การแก้ไขไฟล์คอนฟิก, การอัปเดตความปลอดภัย หรือการรันสคริปต์ต่างๆ บนระบบปฏิบัติการ (OS)]_

### ข้อ 3.2 (6 คะแนน) — *อ้างอิงสถานการณ์*

ในระบบนี้ งานต่อไปนี้ควรใช้ Terraform หรือ Ansible? ตอบพร้อมเหตุผลสั้น ๆ แต่ละข้อ:
- (ก) สร้าง Kubernetes namespace และ persistent volume
- (ข) ติดตั้ง Docker และ kubelet บน server ทั้ง 5 เครื่อง
- (ค) ตั้งค่า cron job สำหรับ backup ฐานข้อมูล

> **ตอบ:** _[(ก) สร้าง Kubernetes namespace และ persistent volume:  
ตอบ: Terraform
เหตุผล: การสร้าง Namespace และ Persistent Volume บน K8s จัดเป็นงานประเภท Infrastructure Provisioning ที่เป็นการสร้างทรัพยากรหลักขึ้นมารองรับระบบ ซึ่ง Terraform มี Kubernetes Provider ที่สามารถจัดการประกาศและควบคุมทรัพยากรเหล่านี้ในรูปแบบ Declarative ได้อย่างมีประสิทธิภาพและติดตามสถานะได้แม่นยำ  
(ข) ติดตั้ง Docker และ kubelet บน server ทั้ง 5 เครื่อง:  
ตอบ: Ansible
เหตุผล: เป็นงานประเภท Configuration Management ที่ต้องเข้าไปดำเนินการภายในระบบปฏิบัติการ (OS Level) ของเครื่องเซิร์ฟเวอร์ที่มีอยู่แล้ว โดย Ansible สามารถ SSH เข้าไปยังเครื่องทั้ง 5 เครื่องพร้อมกันเพื่อสั่งรันคำสั่งติดตั้งแพ็กเกจและจัดการสิทธิ์ต่างๆ ได้รวดเร็วและพร้อมกัน (Automation at Scale)  
(ค) ตั้งค่า cron job สำหรับ backup ฐานข้อมูล:  
ตอบ: Ansible
เหตุผล: เป็นงานจัดการระบบภายในตัวเครื่องเซิร์ฟเวอร์ (OS Configuration/Tuning) Ansible มีโมดูลสำเร็จรูปเฉพาะทาง (เช่น โมดูล cron) ที่สามารถเข้าไปเขียน แก้ไข หรือลบตารางงาน (Crontab) บน OS ได้อย่างแม่นยำและปลอดภัย]_

### ข้อ 3.3 (6 คะแนน)

**Idempotency** คืออะไร และทำไมจึงสำคัญสำหรับ Infrastructure as Code? (อธิบายว่าถ้ารัน script เดิมซ้ำ 2 ครั้งควรเกิดอะไรขึ้น)

> **ตอบ:** _[ความหมายของ Idempotency (การทำงานซ้ำได้ผลลัพธ์เดิม):
คือคุณสมบัติของเครื่องมือ IaC ที่ระบุว่า "ไม่ว่าจะสั่งรันสคริปต์ตัวเดิมซ้ำกี่ครั้งก็ตาม ผลลัพธ์สุดท้ายของระบบปลายทางจะต้องออกมาเหมือนเดิมเสมอ และไม่มีการทำงานซ้ำซ้อนหากระบบอยู่ในสถานะที่ต้องการแล้ว"  
ตัวอย่างสถานการณ์ (ถ้ารันสคริปต์เดิมซ้ำ 2 ครั้ง):  
รันครั้งที่ 1: หากระบบยังไม่มีสิ่งที่เราต้องการ (เช่น สั่งติดตั้ง Docker และสร้างโฟลเดอร์ /data) เครื่องมือ IaC จะทำการติดตั้ง Docker และสร้างโฟลเดอร์นั้นขึ้นมาให้สำเร็จ  
รันครั้งที่ 2: เครื่องมือ IaC จะเข้าไปตรวจสภาพของเซิร์ฟเวอร์ก่อน เมื่อพบว่ามีการติดตั้ง Docker และมีโฟลเดอร์ /data ถูกต้องอยู่แล้ว ตัวสคริปต์จะไม่ทำอะไรเพิ่ม (Skipped) จะไม่มีการลงซอฟต์แวร์ทับซ้อน และไม่มีการสร้างโฟลเดอร์ซ้ำจนเกิด Error  
ความสำคัญต่อระบบ Infrastructure as Code:
ช่วยเพิ่มความปลอดภัยและเสถียรภาพสูงมากในการจัดการโครงสร้างพื้นฐาน เพราะทีม Ops (ซึ่งในทีมมี 2 คน) สามารถนำสคริปต์เดิมมารันซ้ำเพื่อตรวจสอบสถานะของระบบ (Configuration Drift) หรือใช้กู้คืนระบบได้ตลอดเวลา โดยไม่ต้องกังวลว่าจะไปทำลายระบบเดิมที่ทำงานอยู่ หรือทำให้ระบบเกิดข้อผิดพลาด (Human Error) จากการรันคำสั่งซ้ำซ้อน]_

## ส่วนที่ 4: Kubernetes (20 คะแนน)

### ข้อ 4.1 (6 คะแนน)

อธิบายบทบาทของ **Pod**, **Deployment**, และ **Service** ใน Kubernetes และความสัมพันธ์ระหว่างทั้งสาม

> **ตอบ:** _[Pod: หน่วยย่อยที่สุดใน Kubernetes ที่ใช้รัน Container (ในที่นี้คือรันแอป Python Flask) โดย Pod จะทำหน้าที่จัดเตรียมเน็ตเวิร์กและพื้นที่จัดเก็บข้อมูลร่วมกันภายในเครื่องจำลองนั้นๆ  
Deployment: ตัวควบคุม (Controller) ที่ทำหน้าที่ดูแลจัดการวงจรชีวิต (Lifecycle) ของ Pod เช่น การกำหนดจำนวนชุดของ Pod ที่ต้องการให้รัน (Replicas), การสั่งขยายระบบ (Scaling), และการควบคุมกระบวนการอัปเดตเวอร์ชันซอฟต์แวร์  
Service: ตัวทำหน้าที่เป็น Load Balancer และกำหนด IP คงที่ (Static IP/DNS) ให้กับกลุ่มของ Pod เนื่องจาก Pod ภายใน K8s เกิดดับได้ตลอดเวลาและเปลี่ยน IP เสมอ Service จึงทำหน้าที่เป็นประตูหน้าคอยรับ Request แล้วกระจายโหลดส่งต่อไปยัง Pod ที่ทำงานอยู่เบื้องหลังอย่างถูกต้อง  
Deployment เป็นผู้สั่งสร้างและคอยควบคุมจำนวน Pod ให้เป็นไปตามเป้าหมาย (เช่น สั่งให้มี Flask แอป 3 Pod) จากนั้น Service จะใช้ระบบป้ายกำกับ (Labels/Selector) ชี้เป้าไปยังกลุ่มของ Pod ที่สร้างโดย Deployment นั้น เพื่อเปิดช่องทางให้ทราฟฟิกจากภายนอกหรือแอปพลิเคชันอื่นวิ่งเข้ามาใช้งานได้โดยไม่หลงทาง]_

### ข้อ 4.2 (7 คะแนน)

**Readiness probe** กับ **Liveness probe** ต่างกันอย่างไร? ถ้าแอปตั้ง liveness probe ผิด (เช่น ตรวจถี่เกินไป หรือ timeout สั้นเกิน) จะเกิดปัญหาอะไร?

> **ตอบ:** _[Readiness Probe (พร้อมให้บริการหรือไม่): ตรวจสอบว่าแอปพลิเคชันภายใน Pod "พร้อมรับทราฟฟิก (Request) จากผู้ใช้งานหรือยัง" (เช่น ตรวจสอบว่าเริ่มต้นระบบและเชื่อมต่อฐานข้อมูล MySQL สำเร็จแล้ว) หากตรวจสอบไม่ผ่าน K8s จะดึง Pod นั้นออกจาก Service ชั่วคราวเพื่อไม่ให้ผู้ใช้เข้าใช้งานระบบที่ยังไม่พร้อม แต่จะไม่มีการรีสตาร์ท Pod  
Liveness Probe (ยังมีชีวิตอยู่หรือไม่): ตรวจสอบว่าแอปพลิเคชันยังทำงานเป็นปกติหรือค้าง (Deadlock) ไปแล้ว หากตรวจสอบไม่ผ่าน K8s จะถือว่าคอนเทนเนอร์นั้นตายแล้ว และจะทำการ Kill เพื่อรีสตาร์ท (Restart) คอนเทนเนอร์ใน Pod นั้นใหม่ทันที เพื่อกู้คืนระบบให้อัตโนมัติ  
หากแอปพลิเคชันเกิดจังหวะที่ประมวลผลหนักชั่วคราว หรือใช้เวลาสตาร์ทระบบตอนแรกนาน แต่ตั้งเวลาตรวจสอบถี่และสั้นเกินไป K8s จะเข้าใจผิดว่าแอปพลิเคชันค้าง/ตาย และจะส่งคำสั่ง Kill เพื่อรีสตาร์ท Pod วนไปเรื่อยๆ ไม่จบสิ้น เกิดสถานะ CrashLoopBackOff ส่งผลให้ระบบล่มถาวร ทั้งๆ ที่แอปพลิเคชันอาจจะแค่ต้องการเวลาโหลดข้อมูลเพิ่มอีกเล็กน้อยเท่านั้น]_

### ข้อ 4.3 (7 คะแนน) — *อ้างอิงสถานการณ์*

ระบบเดิม deploy แล้วล่ม 15–20 นาที Kubernetes ทำ **zero-downtime deployment** ได้อย่างไร (อธิบายกลไก rolling update)? และถ้า deploy เวอร์ชันใหม่แล้วพบ bug จะ **rollback** อย่างไร?

> **ตอบ:** _[กลไกการทำ Zero-Downtime Deployment (Rolling Update):  
แทนที่จะปิดแอปเก่าทั้งหมดแล้วเปิดแอปใหม่พร้อมกัน (ซึ่งทำให้ระบบเดิมล่ม 15-20 นาที) กลไก Rolling Update ของ K8s จะทำงานแบบค่อยเป็นค่อยไป โดย Deployment จะสร้าง Pod เวอร์ชันใหม่ขึ้นมาขนานกับเวอร์ชันเดิมทีละตัว  
เมื่อ Pod ใหม่ผ่านการตรวจจาก Readiness Probe ว่าพร้อมทำงาน K8s จึงจะเริ่มสลับทราฟฟิกผู้ใช้งานบางส่วนมายังเวอร์ชันใหม่ แล้วค่อยๆ ทยอยลบ Pod เวอร์ชันเก่าทิ้งทีละตัว กระบวนการนี้จะทำวนไปเรื่อยๆ จนกว่าจะเปลี่ยนครบทุก Pod ส่งผลให้ผู้ใช้งานภายนอกไม่รู้สึกถึงความติดขัดและระบบไม่ล่มเลย (Zero-Downtime)  
กระบวนการทำ Rollback เมื่อพบ Bug:  
หากเวอร์ชันใหม่ที่ deploy ขึ้นไปมีบั๊ก ทีม Ops สามารถสั่งการย้อนกลับระบบได้อย่างรวดเร็วผ่านคำสั่ง kubectl rollout undo deployment/<deployment-name>
ตัว Kubernetes จะย้อนกระบวนการ (Reverse Process) โดยค่อยๆ ดึง Pod เวอร์ชันก่อนหน้ากลับมารัน และทยอยทำลาย Pod เวอร์ชันที่มีบั๊กทิ้งไปในรูปแบบ Rolling Update ทำให้ระบบกลับมาทำงานบนเวอร์ชันที่ปลอดภัยได้อย่างเสถียรและไร้รอยต่อเช่นเดียวกัน]_

## ส่วนที่ 5: Monitoring (15 คะแนน)

### ข้อ 5.1 (8 คะแนน)

**Prometheus** และ **Grafana** ทำหน้าที่ต่างกันอย่างไร? และสำหรับเว็บแอปนี้ ควรเก็บ metric อะไรบ้างอย่างน้อย 3 ตัว พร้อมเหตุผล

> **ตอบ:** _[Prometheus: ทำหน้าที่เป็นผู้เก็บรวบรวมข้อมูล (Time-Series Database) โดยจะใช้วิธีวิ่งไปดึงข้อมูล (Pull) ตัวเลขชี้วัด (Metrics) ต่างๆ จากแอป Flask, MySQL และตัวเครื่อง K8s มาบันทึกไว้ตามช่วงเวลา พร้อมทั้งมีระบบตั้งกฎแจ้งเตือน (Alertmanager)  
Grafana: ทำหน้าที่เป็นส่วนแสดงผลภาพ (Data Visualization) โดยจะเชื่อมต่อเพื่อดึงข้อมูลตัวเลขจาก Prometheus มาแปลงเป็นกราฟ, ตาราง หรือแดชบอร์ด (Dashboard) สรุปผลที่สวยงาม เข้าใจง่าย เพื่อให้ทีม Dev และ Ops มอนิเตอร์ระบบได้สะดวก    
3 Metrics ที่ควรจัดเก็บพร้อมเหตุผล:  
HTTP Request Latency / Duration (ความเร็วในการตอบสนองของ Flaskแอป): เหตุผล: เพื่อดูประสิทธิภาพของโค้ดและรับรู้ว่าผู้ใช้งานกำลังเจอปัญหาเว็บอืด เว็บค้าง หรือโหลดช้าเกินไปหรือไม่  
Pod Memory / CPU Usage (การใช้ทรัพยากรบน K8s): เหตุผล: เฝ้าระวังไม่ให้เกิดปัญหา Resource ขาดแคลนจนนำไปสู่เหตุการณ์ Container ถูกทำลายทิ้งเพราะหน่วยความจำเต็ม (OOMKilled)  
MySQL Active Connections (จำนวนการเชื่อมต่อของฐานข้อมูล): เหตุผล: เนื่องจากเปลี่ยนจากแอปเดี่ยวมาเป็น K8s ที่ขยาย Pod ได้หลายตัว ทุกตัวจะวิ่งรุมเข้าหา MySQL เครื่องเดิมพร้อมกัน จึงต้องมอนิเตอร์ไม่ให้ Connection เต็มขีดจำกัดจนแอปเชื่อมต่อฐานข้อมูลไม่ได้]_

### ข้อ 5.2 (7 คะแนน)

ทีม ops อยากรู้ปัญหา **ก่อน** ผู้ใช้ร้องเรียน ควรตั้ง alert อะไรบ้าง? ยกตัวอย่าง alert 2 ข้อ พร้อม condition และเหตุผลว่าทำไม alert นี้สำคัญ

> **ตอบ:** _[Alert 1: Pod Memory Usage Near Limit  
Condition: container_memory_usage_bytes / container_spec_memory_limit_bytes > 0.85 for 5m (การใช้งาน Memory ของ Container สูงเกิน 85% ของ Limit ติดต่อกันเป็นเวลา 5 นาที)  
เหตุผลความสำคัญ: เตือนเพื่อให้ Ops ทราบว่าแอป Flask อาจเกิดปัญหา Memory Leak หรือรับโหลดสูงเกินไป Ops จะได้เข้าตรวจสอบหรือปรับขนาด (Scale) เพิ่มเติมได้ทันท่วงทีก่อนที่ Memory จะแตะ 100% จนระบบสั่ง Kill Pod ทิ้ง
