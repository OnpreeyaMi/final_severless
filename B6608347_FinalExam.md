## ส่วนที่ 1: Git & CI/CD (25 คะแนน)

### ข้อ 1.1 (8 คะแนน) — *อ้างอิงสถานการณ์*

ทีม dev 4 คนเดิม push เข้า `main` พร้อมกันทำให้ conflict บ่อย ออกแบบ **Git branching strategy** ให้ทีมนี้ อธิบาย: branch ที่มี, วัตถุประสงค์ของแต่ละ branch, และ flow เมื่อจะ deploy ขึ้น production

> **ตอบ:** _[1.main เป็นbranchหลักที่เสถียร โค้ดพร้อมขึ้นdeploy ล็อกไว้ไม่ให้ใครpushตรงๆ  
2.feature/* branch ชั่วคราวที่แต่ละคนจะแยก    flow [ feature/* branch ] ──(สร้าง PR & ตรวจสอบ)──> [ main branch ] ──(Jenkins Build & Deploy)──> [ K8s Production ]ออกไปจาก main เพื่อแตกงานฟีเจอร์ใหม่หรือแก้บั๊กของตัวเอง]_

### ข้อ 1.2 (10 คะแนน)

CI/CD pipeline ของระบบนี้ควรมี stage อะไรบ้างเรียงตามลำดับ (ตั้งแต่ developer push code จนถึง deploy)? อธิบายว่า **แต่ละ stage ทำอะไร** และ **ทำไมต้องเรียงลำดับนี้**

> **ตอบ:** _[ออกแบบ Pipeline โดยใช้หลักการ "Fail-Fast"   
  Stage 1: Checkout Code Jenkins จะทำการดึงซอร์สโค้ดล่าสุดจาก Git Repository หลังจากได้รับสัญญาณ Webhook เมื่อมีการ Merge โค้ดเข้าสู่สายหลัก  
  ทำไมต้องอยู่ลำดับนี้: เป็นจุดเริ่มต้นที่จำเป็นที่สุด เพราะหากไม่มีตัวโค้ดล่าสุด ก็จะไม่สามารถนำไปประมวลผลหรือทดสอบในขั้นตอนอื่นได้เลย
  Stage 2: Code Linting & Static Code Analysis ใช้เครื่องมือตรวจสอบความสะอาดของโค้ด Python เช่น Flake8 หรือ Black เพื่อเช็ก Syntax, ตรวจสอบ Format โค้ดให้เป็นมาตรฐานเดียวกัน  
  ทำไมต้องอยู่ลำดับนี้: เป็นขั้นตอนที่ทำงานเร็วที่สุด (ใช้เวลาเป็นวินาที) และไม่ต้องอาศัยสภาพแวดล้อมจำลองใดๆ หาก Dev พิมพ์โค้ดผิดฟอร์แมตหรือเขียน Syntax พัง ระบบจะแจ้งเตือนและหยุดทำงานทันทีโดยไม่ต้องเสียเวลาไปรันเทสใหญ่  
  Stage 3: Unit Testing เพื่อตรวจสอบ Logic ภายในแอปพลิเคชัน Flask ว่าฟังก์ชันต่างๆ ยังทำงานได้ถูกต้องตามที่ออกแบบไว้   
  ทำไมต้องอยู่ลำดับนี้: ทำหลังจากจัดฟอร์แมตโค้ดเสร็จ เพื่อพิสูจน์ว่า Logic ของแอปพลิเคชันไม่มีบั๊กและทำงานได้ถูกต้องตามความต้องการ ก่อนที่จะนำโค้ดนี้ไปทำเป็น Container Image ในขั้นตอนต่อไป  
  Stage 4: Build Docker Image ทำการประกอบร่าง (Package) แอปพลิเคชัน Flask, Dependencies ทั้งหมด (จาก requirements.txt) และสภาพแวดล้อมที่จำเป็นขึ้นมาเป็น Docker Image ตามคำสั่งที่เขียนไว้ใน Dockerfile  
  ทำไมต้องอยู่ลำดับนี้: ควรทำหลังจากที่มั่นใจว่าโค้ดไม่มีบั๊กพื้นฐานแล้ว (ผ่าน Stage 2 และ 3) เนื่องจากกระบวนการ Build Image ค่อนข้างใช้เวลาและทรัพยากรเครื่อง (Compute Resource) หากโค้ดพังตั้งแต่ Logic จะได้ไม่ต้องเสียเวลา Build ให้เปลือง Resource  
  Stage 5: Security Scanning (Vulnerability Scan) ใช้เครื่องมือเช่น Trivy หรือ Clair สแกนเข้าไปใน Docker Image ที่เพิ่ง Build เสร็จ เพื่อตรวจหาช่องโหว่ด้านความปลอดภัย (Vulnerabilities) ในระดับ OS Packages และ Library ต่างๆ ที่ดึงมาใช้  
  ทำไมต้องอยู่ลำดับนี้: ต้องทำทันทีหลังจากที่ได้ Docker Image มา เพื่อคัดกรองความปลอดภัยให้มั่นใจ 100% ว่าจะไม่มีโค้ดหรือแอปพลิเคชันที่มีช่องโหว่อันตรายหลุดรอดขึ้นไปรันบน Production Server  
  Stage 6: Push Image to Registry นำ Docker Image ที่ผ่านการตรวจสอบความปลอดภัยเรียบร้อยแล้ว อัปโหลด (Push) ไปเก็บไว้ที่ Private Docker Registry ส่วนกลางขององค์กร โดยติดแท็กเวอร์ชันอย่างชัดเจน (เช่น เลข Git Commit SHA) เพื่อให้ Kubernetes สามารถเข้ามาดึงไปใช้ได้  
  ทำไมต้องอยู่ลำดับนี้: เพราะนี่คือขั้นตอนสุดท้ายของการเตรียม Artifact (ตัวแอปที่พร้อมรัน) ก่อนที่จะสั่ง Deploy จริง Image ต้องถูกเก็บไว้ในที่ที่ปลอดภัยและเข้าถึงได้จากภายนอกก่อน  
  Stage 7: Deploy to Kubernetes Cluster (Production) Jenkins จะส่งคำสั่งไปอัปเดตไฟล์คอนฟิก (Kubernetes Manifests) เพื่อเปลี่ยนเวอร์ชัน Image เป็นตัวล่าสุด และสั่งให้ K8s ทำการ Rolling Update ค่อยๆ ทยอยเปลี่ยน Pod ของ Flask ทีละตัว เพื่อสลับรับทราฟฟิกของผู้ใช้อย่างต่อเนื่องแบบไร้รอยต่อ (Zero-Downtime)  
  ทำไมต้องอยู่ลำดับนี้: เป็นขั้นตอนสุดท้ายที่มีความสำคัญที่สุดและใช้ความเสี่ยงสูงที่สุด จึงถูกวางไว้หลังสุดหลังจากที่ตัวแอปพลิเคชันผ่านการทดสอบทั้งเรื่อง คุณภาพ ความถูกต้อง และความปลอดภัยมาอย่างครบถ้วนแล้ว เพื่อให้มั่นใจได้ว่าระบบใหม่ที่จะขึ้นไปรันจะไม่ทำให้ระบบล่มอย่างแน่นอน]_

## ส่วนที่ 2: Docker & Containerization (20 คะแนน)

### ข้อ 2.1 (7 คะแนน)

อธิบายว่า **container ต่างจาก virtual machine (VM) อย่างไร** และการ containerize แอป Flask + MySQL ช่วยแก้ปัญหา "บนเครื่องผมรันได้แต่บน server รันไม่ได้" ได้อย่างไร

> **ตอบ:** _[ความแตกต่างระหว่าง Container กับ Virtual Machine (VM):
Virtual Machine (VM): เป็นการจำลองในระดับฮาร์ดแวร์ (Hardware Virtualization) ผ่าน Hypervisor โดยแต่ละ VM จะต้องมี Guest OS (ระบบปฏิบัติการเต็มรูปแบบ) เป็นของตัวเอง ทำให้กินทรัพยากร (CPU, RAM, Disk) สูงมาก บูตระบบช้า (เป็นนาที) และมีขนาดใหญ่ (หลาย GB)
Container: เป็นการจำลองในระดับระบบปฏิบัติการ (OS Virtualization) โดยแชร์ Host OS Kernel ร่วมกัน และใช้ฟีเจอร์ของ Linux (เช่น Namespaces, Cgroups) ในการแยกสภาวะแวดล้อม (Isolation) ทำให้ไม่ต้องมี Guest OS ส่งผลให้กินทรัพยากรต่ำมาก บูตเร็วในระดับวินาที และมีขนาดเล็กมาก (ระดับ MB)   
  การแก้ปัญหา "บนเครื่องผมรันได้แต่บน server รันไม่ได้" ด้วย Portability:
ปัญหานี้มักเกิดจากสภาพแวดล้อม (Environment Mis-match) เช่น เวอร์ชันของ Python หรือ Library ในเครื่อง Dev ไม่ตรงกับบน Server
การทำ Containerize จะทำการมัดรวม (Package) ตัวแอปพลิเคชัน Flask, ซอร์สโค้ด, Runtime (Python 3.11), Dependencies ทั้งหมด (requirements.txt) และ Environment Variables ที่จำเป็น เข้าไว้ด้วยกันใน Docker Image เพียงหนึ่งเดียว ทำให้มั่นใจได้ว่า สภาพแวดล้อมที่แอปพลิเคชันใช้จะถูกล็อกไว้คงที่ ไม่ว่าจะนำ Image นี้ไปรันบนคอมพิวเตอร์ของ Dev คนไหน หรือบน On-premise Server เครื่องใดก็ตาม ระบบจะทำงานได้ผลลัพธ์เหมือนกัน 100%]_
